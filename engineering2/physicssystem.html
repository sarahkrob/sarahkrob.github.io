What the project does:


My project is, in writing, a physics system, but actually serves more as a simple movement system that can be used with key presses to control a player character. It allows for rotation and linear movement of an object with application of acceleration (friction/drag) that will gradually slow and stop movement. It also allows the user to apply bounds to the movement of the object in all directions, in order to constrain it to a certain area. 


The user can read in a file that contains the dimensions (width/height/depth) of the object that the physics system is being used with as well as bindings for all keys that would impact movement specifically carried out by the physics system (up, down, left, right, forward, back, and rotation left and right). If a file is not read in, then the object containing the physics system will not be moveable, but will use the physics object in order to get the position and rotation for rendering.


The public interface is fairly simple and mostly consists of setters. The only other functions a user should need are the constructor with parameters, which only actually requires the user to enter an initial position and rotation for the object-- all other parameters have default values-- and the UpdatePhysics function. This should be called wherever one would like to update the physics of an object.

How to Use:


You will need to download the .zip attached at the top of the write-up. This should contain a folder called NewPhysics.

This should be copied to your Engine directory, and then added as an Existing Project in Visual Studio. In the case of my project, I did not need any references to other projects. If there is an issue in this regard, please let me know.


You will need to add NewPhysics as a reference in whatever project your GameObject class is stored, and maybe in your MyGame project, depending on how things are implemented. That said, the minimum you should need to do to use it is add a cPhysicsObject as a member variable in your GameObject, and then construct that object when you construct your GameObject.

Conclusion:


This semester I learned a lot about platform-agnostic programming, and though my system didn’t end up needing platform-specific implementations, I kept that in mind, as well as ease for the user, while creating it. I also had never really worked with static libraries before this, so you could consider that something I took away from the class. Most notably, I used what I learned to create a new Builder to be used with .lua files that the user can create, convert those to .bin files, and then read them in my physics system. This makes my physics system, I hope, as user friendly as possible and allows them to remove things like reading user input for movement from their MyGame file.


I struggled with this project mostly in regards to my original ideas for what I wanted to do. I learned a bit too late that one of my initial ideas, applying impulses to objects at various points in the scene, would require basically implementing things needed for a collision system in order to process the faces/normals of where said impulse was hitting the object. Therefore, though I was able to apply a movement force to an object on a key press, I could not do this from a specific point in the scene. This makes the existence of torque in my project, as well as the ApplyForce method, pretty obsolete, and I recommend people not use them. I pivoted in the last week of the project to focus on making a simple movement system that would be usable for simple, arcade-style games instead and would allow users to input their own key bindings for the movement the system manages. Because of that pivot, I also added the ability to set movement bounds at the end of the assignment, to make it more usable for an entire project. I’m happy with what I produced in terms of what I was aiming for at the end.
