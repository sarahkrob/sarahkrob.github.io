My implementation still stores the same amount of data for storing meshes and effects as the last assignment, 232 bytes for the struct with one additional matrix now being used for the draw struct. I didn’t change how information was stored in Graphics at all, and am not sure if I needed to. 


One thing to note that I did change was moving my mesh/effect/background information into the struct itself, and not storing it in the constantData_frame sFrame struct as I was doing before. I misunderstood the instructions on the last assignment, and that’s since been adjusted. I also did some other refactoring of my code since last time that I’m much happier with-- the main change is swapping the mesh and effect submissions to be done in the same method. I also moved my created meshes to be stored as member variables for the game on startup, which makes using them much easier. 


An additional note: my game still only runs on x64 debug. I have not been able to figure out the OpenGL shader issue from last time, and my x64 release opens and immediately hangs. I’ve decided for now to continue working with just x64 debug for the sake of completing the assignments, but hope the others can be fixed soon, even though I have no idea how to.


For this assignment, I created a separate gameObject and camera class, with their own methods. I actually feel now that camera should be a child of gameObject, as they share several methods, but have not yet made that adjustment. I attempted to implement extrapolation for the movement in these classes, but I actually think it made my movement look a bit worse. The change was minimal, if anything. I’m not sure if this means my game was already set up correctly to begin with, or if I’m calling the position update too often so there’s no need to extrapolate, or what-- either way, the use of extrapolation is so that in the time it’s taking to return to the program and get the updated drawing information for the object, the movement doesn’t look jerky. For example, if you don’t call the draw function every frame, but need to move the player from one place to another, you can extrapolate where they might be each frame based on the time that’s passed for that frame.


That’s the only major confusion I had with this assignment. Otherwise, I was pleased that everything seems to be working smoothly, aside from preexisting issues. I do think I may need to adjust my implementation in the future to be more generalized for an actual game, as it’s pretty hardcoded for this specific assignment right now, but the basics are in and I don’t have any major bugs that I know of.
