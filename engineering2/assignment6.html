In this assignment we changed our projects to include a MeshBuilder project, which allows users to create separate mesh files that can then be read into the program via path, rather than hardcoding in the vertices and indices for every mesh. This is a huge advantage because it A: means anyone can make a mesh because the format of the file is pretty simple and straightforward, B: means meshes can be read in from other programs, like we do in Assignment 7, and C: makes issues in the vertices easy to debug. 


In setting up my mesh file format, I tried to pick the way that was simplest for me to understand and contained the least amount of data. I found using a dictionary to be a little counterintuitive when I feel like with very little searching around, nearly anyone who would ever want to make a mesh would understand {x, y, z} format. This goes double if I were to simply include a comment at the top saying what the format is. As for the large array of vertices, it becomes more readable in Assignment 7 (already completed) where a comment is added following every set of coordinates and they’re all spaced out one per line. This is simply how I originally had it formatted in my program to save space when I was hardcoding, but it’s very flexible and I think easy to understand with comments added. Complications might arise when adding more complex data, like color, but I don’t really think it would be that hard, for example, if a third array was added that mirrored the vertex one position-wise.
