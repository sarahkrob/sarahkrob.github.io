NOTE: I would like to change this, as I mention below.


The reason that we need to cache information and not render it immediately is so that our application can be multithreaded, with cached data being used depending on which thread is currently active (information from the application thread is submitted to the render thread when drawing the frame, so that these things can occur concurrently). If there was no caching, there would be a chance that the application thread could wipe out data before the render thread is able to use it, causing the program to crash.

My total memory for rendering frames is 480 bytes. Each struct is 240 on its own, and given that there are two of them, that’s the final count. I have two arrays that store four pointers each in my structs, allowing four mesh-effect pairs to be stored. This number was chosen arbitrarily, as I only render two. I also added two ints, so that the current index of each of those can be stored and used. 


Now for how the project actually went: things were totally smooth sailing except for the submitting meshes portion of the assignment. My graphics data, backgrounds and reference counting were all working totally fine before adding this part. In addition, I was able to implement user input without much issue, as I did this in Assignment 1. The way I have it set up currently is to change a state enum based on what key is pressed, and to use that to determine what meshes to use in the submit method. 


Issues arose when it came time to start submitting meshes and effects. I struggled a bit with adjusting my existing code to fit the new caching system, but ultimately decided on what data to store and a design that I thought was correct. Evidently, given that three of my four builds don’t work, I was incorrect in some way. Currently, my game works correctly on a x64 debug build and nothing else (as a result, a debug build is what’s attached to download on the webpage for now). My current problems are that my OpenGL build fails to create effects or meshes (both are broken, as I tested in two different orders of creation) due to a 1282 error, which is an access violation. My belief is that this has something to do with the glContext not being registered correctly, as the data for my class is, as far as I know, instantiated correctly. I have been unable to debug this as much as I have tried, and a few other people I’ve spoken to are also having this issue. Asking Tony, he also had no idea what could be causing it. My only other guess was that it has to do with my effect class not being 16-byte aligned in size, but I tried buffering the size up to 32 on win32 to no avail. 


In addition, my game fails on release builds for both platforms, despite working totally fine on x64 debug. On x64 this appears to be due to an access violation issue that either shows up related to vertexes or throws me into disassembly. Occasionally, my game will start up and immediately freeze without throwing an error, others it will crash due to said access violation issue. On win32, the issue on release looks to have something to do with the scopeGuards currently used in Load functions throughout the project. Though they worked totally fine on prior assignments and before implementing the submission of meshes and effects, they now fail to set the referenced shaders/effects/meshes to anything, leaving a null reference that crashes the game.


Given that I’m also unsatisfied with my design for how meshes and effects are submitted (I misunderstood the instructions and wrote two separate functions and, while functionally they’re the same as having one large function, I don’t think they’re very user-friendly and would like to change that) I believe my best option at this point is to back up to my last working commit and reimplement this part of the assignment with my new understanding of what needs to be set up. I’m hoping I’ll manage to fix it in the process of making things cleaner. I unfortunately did not have time to do this before the assignment was due due to IRL circumstances, but I’m hoping to be able to get it working in order to implement the next assignment.
